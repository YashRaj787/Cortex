# Feature Request Management Process

This document maintains a database of feature requests with details on their status, frequency, and prioritization.

## Feature Request Template

### FR-<nnn>

**Status:** (Open/Collecting Feedback/Planned/Implemented/Rejected)

**Request Count:** <count>

**Frequency:** (Rare/Occasional/Frequent)

**User Type:** (Student/Developer/Professional/Other)

**Description:**
<detailed description>

**Problem Being Solved:**
<specific problem this solves>

**Requested By:**
<user name/identifier>

**Date Requested:**
<yyyy-mm-dd>

**Priority:**
(Low/Medium/High)

**Decision:**
(Collecting Feedback/Planned/Rejected/Implemented)

**Notes:**
<any additional context or discussions>

---

### Frequency Definitions

**Rare**
* Requested by 1 user

**Occasional**
* Requested by 2-3 users

**Frequent**
* Requested by 4+ users

---

### User Type Definitions

**Student**
- Users learning to code or manage notes

**Developer**
- Professional code managers or developers

**Professional**
- Business users or non-developers

**Other**
- Any other user type

---

### Workflow

Requested → Collecting Feedback → Under Review → Planned → Implemented
                                    ↓
                                    Rejected

---

### Prioritization Rules

1. Do not build a feature because one user asks
2. Consider building when:
   - Request frequency increases significantly
   - Multiple different user types request it
   - Feature aligns with the product vision

---

### Review Cadence

- Review feature requests every 2 weeks

---

### Success Criteria

- Feature decisions are driven by patterns, not individual opinions
- Regular collection and review of feature requests
- Clear criteria and communication for feature decisions
